## Scaffold notes (this build)

- **Fixture is synthetic.** `tests/fixtures/bicep_curl_1.json` is generated by
  `scripts/make_fixture.py` (8 reps, 15 Hz, cosine-eased CON/ECC with holds and
  jitter) so the harness and Gate A wiring work today. **The gates only mean
  something against a real human-labelled capture** — swap it in (same JSON
  shape) before trusting Gate A numbers. The synthetic labels were produced by
  the generator's own phase model, so a Gate A pass against the synthetic
  fixture proves the *plumbing*, not the *tuning*.
- **Ingest tests are Postgres-gated, by design.** The ORM uses Postgres-native
  JSONB/UUID + a named schema. Rather than down-cast to SQLite (which would let
  tests pass against a schema that differs from production), `test_ingest.py`
  skips unless `GYMBOX_TEST_DB` is set. DSL/signal/ETag tests are pure-unit and
  always run.
- **`pipeline/rep.py` front-end is wired before the `NotImplementedError`** so
  the exception is reachable and the Gate A test xfails cleanly (rather than
  throwing an unrelated error earlier). The signal layer (`signal.py`) and
  `evaluate_phase` are concrete and unit-tested.
- **Swift `DSLInterpreter.interpret` traps (`fatalError`)** until ported — Gate B
  target. `evaluatePhase`, `conditionHolds`, `DynamicBands`, and the whole
  `Signal` front-end are provided and mirror the Python.
- **Keypoint registry is duplicated** in `gymbox/dsl/keypoints.py` and
  `Keypoints.swift` and must stay byte-for-byte aligned. A Swift test spot-checks
  the indices db_curl relies on (right_wrist=16, elbow=14, shoulder=12). Consider
  generating one from the other at Step 1 to remove the duplication risk.
- **S-G edge handling**: the Swift port uses scipy-style `mode='interp'`
  (polynomial fit on the edge window, evaluated at the offset). Confirm this
  matches whatever `signal.py` does at the boundaries when chasing the last
  couple of % for Gate B — boundary frames are exactly where cross-language
  drift hides, which is why Gate B allows ±2-frame rep boundaries.

## Things to watch

- The proto reserves `PoseFrame` fields 9–15 and a signal-type vocabulary for
  future carriers (hand landmarks, objects, IMU…). Don't assign those numbers.
- `model_spec` stays `null` through MVP-α. The interpreter checks
  `if model_spec is not None` and treats null as a no-op — keep that branch free
  of model-loading overhead.

## db_curl fitted to real data (2026-06-14)

- **Phase convention changed to follow the data** (owner decision): bottom hold →
  `ISO_UNLOADED`, top pause → `RESET`, `ISO_LOADED` unused. db_curl phase rules +
  `window_frames` (7→9) and `abs_v` (0.04→0.08) were set by the offline fitter
  (`server/scripts/fit_db_curl.py`) to maximise per-side Gate A on the labelled
  videos. architecture.md §10 updated; synthetic fixture + Gate B oracle
  regenerated to the new convention (both still pass).
- **Open: phase agreement is 0.759 per-side, below the 0.85 bar.** Rep counting is
  solved (mean err 0.08). The remaining gap is phase-boundary disagreement (human
  label jitter at CON/ECC/hold transitions; RESET-at-top vs end-of-CON fuzziness).
  Levers not yet tried: tune `min_amplitude`/`prominence_frac`, a `band_frac`
  grammar field (currently a hard-coded 0.25 constant in `rep.py` — not tunable
  via the spec), `sign_changed_within_ms` transition smoothing, boundary-tolerant
  scoring, and simply more labelled data.
- **Alternating arms:** db_curl tracks `right_wrist`; ~45% of these videos are
  left-arm reps. Evaluation is **per-side** (track the active wrist). A
  side-aware signal (or per-side spec selection on device) is a real MVP-β item.
- **Both-arm detection / "active arm" gating (MVP-β).** To track both arms
  label-free, run the interpreter independently per wrist (the viz does this).
  But the *resting* arm over-counts (e.g. Bicep Curl 8: left wrist → 24 reps vs
  ~13) because the idle arm isn't perfectly still. Doing both arms correctly needs
  an **active-arm gate** — only credit a rep on the arm that is actually working.
  This maps to the reserved `active`/`inactive` annotation layers (l7/l8, already
  in the label schema) and/or a `composite` signal (architecture.md §11.3). The
  per-frame phase tint is fine; it's the rep *count* on the idle arm that inflates.
- **`band_frac` is not in the DSL grammar** (hard-coded 0.25 in
  `DynamicBands`). If band tuning matters for the fitter, promote it to a spec
  field first — don't hard-code spec values.
